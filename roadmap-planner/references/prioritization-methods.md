# Roadmap 优先级方法

选择一种足够简单、能解释取舍的方法。不要为了精确而制造虚假的数字。

## 推荐默认方法：Value / Urgency / Confidence / Effort

适合大多数产品规划。

| 维度 | 说明 | 评分建议 |
| --- | --- | --- |
| Value 价值 | 对用户目标和业务目标的贡献 | 1-5 |
| Urgency 紧迫度 | 是否有窗口期、承诺、风险或竞争压力 | 1-5 |
| Confidence 信心 | 证据是否充分，问题和方案是否清楚 | 1-5 |
| Effort 工作量 | 产品、设计、研发、测试、运营复杂度 | 1-5，分数越高越复杂 |

可用公式：

```text
Priority Score = (Value + Urgency + Confidence) / Effort
```

评分只能辅助讨论，最终排序要结合依赖、战略主题和资源节奏。

## RICE

适合增长、实验、功能池较多的场景。

- Reach：影响多少用户或次数
- Impact：单个用户/业务影响有多大
- Confidence：信心程度
- Effort：工作量

注意：不要编造 Reach。没有数据时用相对等级或标记待确认。

## MoSCoW

适合 MVP 范围收敛。

- Must：没有它就无法验证核心价值
- Should：重要但可在首版后补充
- Could：体验增强
- Won't：本阶段明确不做

## Kano

适合体验型产品或用户满意度规划。

- 基础型：没有会不满，有了不一定惊喜
- 期望型：越好越满意
- 兴奋型：超预期亮点

## Dependency-first Sequencing

适合平台、数据、权限、基础设施依赖强的 roadmap。

优先处理：

- 阻塞多个后续需求的基础能力
- 数据模型和权限模型
- 关键集成与接口
- 风险高、需要提前验证的技术路径

## 输出建议

优先级输出应包含：

- 排名
- 机会/项目名称
- 所属主题
- 价值判断
- 紧迫度或窗口期
- 信心来源
- 工作量/复杂度
- 依赖
- 推荐阶段
- 取舍说明
