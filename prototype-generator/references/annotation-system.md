# 原型标注系统

标注系统用于把 PRD 需求、页面模块、交互规则和验收要点连接起来，让产品、设计、研发和测试可以围绕同一处界面理解需求。

## 1. 标注原则

- 每个关键页面模块都应绑定一个或多个需求 ID。
- 点击页面模块时，右侧标注面板应显示对应说明。
- 点击标注项时，页面中的对应模块应高亮。
- 搜索标注时，应能按需求 ID、模块名称、页面名称、关键词定位。
- 不能可视化的需求应进入“非可视化需求”列表，并说明原因。

## 2. 标注数据结构

建议使用以下结构。单文件 HTML 中可以内置为 JavaScript 数组；多文件输出时可以保存为 `annotations.json`。

```json
[
  {
    "id": "REQ-001",
    "page": "页面名称",
    "moduleId": "module-xxx",
    "moduleName": "模块名称",
    "type": "功能需求",
    "description": "需求说明",
    "interaction": "交互规则",
    "states": ["默认", "空状态", "错误状态"],
    "acceptance": ["验收标准 1", "验收标准 2"],
    "notes": "假设或待确认问题"
  }
]
```

## 3. 页面模块绑定

每个可标注模块建议包含：

```html
<section
  class="annotated-module"
  data-annotation-id="REQ-001"
  data-module-id="module-search-filter">
  ...
</section>
```

交互要求：

- hover 时显示轻微描边或背景提示。
- click 时选中模块，并打开/更新右侧标注。
- 当前选中模块需要高亮。
- 如果一个模块对应多个需求，标注面板显示需求列表。

## 4. 右侧标注面板

建议包含：

- 搜索框
- 当前选中模块
- 需求 ID
- 功能说明
- 交互规则
- 状态说明
- 权限说明
- 验收标准
- 待确认问题
- 非可视化需求列表

## 5. 标注分类

常见类型：

- 功能需求
- 用户流程
- 交互规则
- 状态说明
- 权限规则
- 数据说明
- 埋点指标
- 验收标准
- 非可视化需求
- 待确认问题

## 6. 搜索与定位

搜索应覆盖：

- 需求 ID
- 页面名称
- 模块名称
- 功能描述
- 状态
- 验收标准

搜索结果点击后：

- 切换到对应页面
- 高亮对应模块
- 更新右侧标注面板

## 7. 非可视化需求

以下内容可能没有直接 UI 表现，但仍应出现在标注中：

- 后端校验
- 数据同步
- 权限审计
- 埋点上报
- 性能要求
- 安全/隐私规则
- 运营配置
- 回滚策略

对每项说明：

- 对应需求 ID
- 为什么不可视化
- 研发/测试需要关注什么
