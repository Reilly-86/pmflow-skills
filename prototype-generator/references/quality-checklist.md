# 原型质量检查清单

用于生成或评审 HTML 交互原型前的最后检查。

## 严重度

- `阻塞 Blocking`：无法进行有效评审或关键交互不可用。
- `重要 Major`：缺少关键页面、流程、状态、需求映射或标注。
- `次要 Minor`：清晰度、一致性、布局或易用性问题。
- `建议 Suggestion`：可选增强。

## 检查项

### PRD 覆盖

- 是否覆盖核心页面和核心流程。
- P0 功能是否都有可视化模块或非可视化说明。
- 关键状态是否被展示或标注。
- 权限和角色差异是否被展示或说明。
- 数据和埋点类需求是否进入标注或非可视化需求列表。

### 交互可用性

- 页面导航是否清楚。
- 弹窗、Tab、筛选、搜索、表单、状态切换是否可操作。
- 操作反馈是否可见。
- 空状态、错误状态、加载态是否能被查看。
- 交互是否符合 PRD 描述，没有额外引入未经说明的业务规则。

### 标注系统

- 页面模块是否能点击查看标注。
- 标注项是否能反向定位页面模块。
- 搜索是否覆盖需求 ID、模块、页面、描述和验收标准。
- 标注中是否包含功能说明、交互规则、状态说明和验收标准。
- 非可视化需求是否单独列出。

### 视觉与布局

- 信息结构是否清晰，适合产品评审。
- 文本是否可读，不溢出、不遮挡。
- 高亮状态是否明显。
- 布局在常见桌面视口下是否稳定。
- 不使用无意义装饰影响需求理解。

### 交付说明

- 是否列出原型假设。
- 是否列出缺失信息和待确认问题。
- 是否说明生成文件和使用方式。
- 是否说明原型不是生产代码。

## 常见修正

- 只有静态页面 -> 增加关键点击、跳转、弹窗和状态切换。
- 只有 UI 没有标注 -> 增加需求 ID、模块绑定和右侧说明。
- 标注无法定位 -> 增加搜索结果点击和模块高亮。
- 缺少异常态 -> 补空状态、错误态、权限不足和提交失败。
- 视觉过度设计 -> 回到清晰、克制、评审友好的信息结构。
