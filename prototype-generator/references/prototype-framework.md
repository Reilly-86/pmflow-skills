# PRD 到原型分析框架

用于从 PRD 或结构化需求中提取可视化原型所需的信息。目标不是重写 PRD，而是建立页面、模块、交互、状态和需求 ID 的映射。

## 1. 输入解析

从 PRD 中提取：

- 产品/功能名称
- 目标用户和角色
- 核心场景
- 页面列表
- 功能模块
- 用户流程
- 功能需求
- 交互规则
- 状态说明
- 权限规则
- 数据对象和字段
- 埋点或指标
- 验收标准
- 风险、假设和待确认问题

如果 PRD 缺少需求 ID，为重要需求生成稳定 ID：

- `REQ-001`：功能需求
- `FLOW-001`：用户流程
- `STATE-001`：状态说明
- `ROLE-001`：权限规则
- `DATA-001`：数据/字段说明

稳定 ID 规则：

- 优先沿用 PRD 中已有的 ID。
- 没有 ID 时，按页面顺序和模块顺序生成，例如 `REQ-P01-M01`。
- 流程可用 `FLOW-001`，状态可用 `STATE-P01-01`，权限可用 `ROLE-001`，数据可用 `DATA-001`。
- 不使用随机 ID。
- 后续修改时保留已有 ID，新增需求追加编号。
- 如果必须改变 ID，说明影响到的标注和模块。

## 2. 页面与模块建模

为每个页面识别：

- 页面名称
- 页面目标
- 入口
- 页面模块
- 主要操作
- 次要操作
- 空状态
- 加载态
- 错误态
- 成功态
- 跳转目标

模块建议包含：

- 模块 ID
- 模块名称
- 对应需求 ID
- 展示数据
- 可执行动作
- 交互反馈
- 权限差异
- 验收要点

## 3. 交互建模

优先实现能帮助评审需求的交互：

- 页面跳转
- Tab 切换
- 筛选和搜索
- 列表选择
- 详情展开
- 弹窗打开/关闭
- 表单输入和校验
- 提交成功/失败反馈
- 状态切换
- 角色/权限视图切换
- Mock 数据变化

不要为了炫技加入 PRD 没有要求的复杂动画或业务逻辑。

## 4. 状态与异常可视化

核心流程至少考虑：

- 默认状态
- 空状态
- 加载状态
- 错误状态
- 成功状态
- 权限不足
- 输入非法
- 重复操作
- 操作不可逆
- 第三方失败或数据同步失败

如果某个状态不会出现在 UI 中，在标注中说明它是后端、数据、运营或测试关注点。

## 5. 原型层级

默认生成评审级原型：

- 能表达信息结构
- 能模拟关键流程
- 能展示关键状态
- 能绑定需求标注
- 不追求高保真视觉
- 不连接真实后端

根据 PRD 判断设备形态：

- 管理后台、SaaS、内部工具：默认桌面 Web
- C 端 App、移动流程：默认移动端布局
- 响应式产品：说明桌面/移动差异
- 不明确时：默认桌面 Web，并标记假设

复杂度控制：

- 复杂 PRD 优先生成核心流程原型，而不是一次覆盖所有页面和所有状态。
- 当页面超过 3 个或交互过多时，先输出页面/模块/标注计划，再生成文件。
- P0 功能优先可视化，P1/P2 可以用标注或后续迭代说明。

## 6. 原型假设

当输入不足时，输出中必须列出：

- 页面结构假设
- 交互行为假设
- 权限假设
- 数据字段假设
- 状态假设
- 需要 PRD 补充的信息
