# 需求分析框架

用于把原始需求拆成可评审、可进入 PRD 的结构化信息。不要机械填满所有项，只保留会影响产品、设计、研发、测试、数据或运营决策的内容。

## 1. 需求来源与背景

- 来源：业务方、用户反馈、客户成功、销售、运营、数据、合规、管理层、内部团队等。
- 触发原因：为什么现在提出，当前有什么问题或机会。
- 当前状态：现有流程、已有功能、人工处理方式、已知限制。
- 影响对象：用户、内部角色、上下游团队、外部系统。

## 2. 目标与成功标准

- 业务目标：效率、转化、留存、收入、成本、合规、交付质量等。
- 用户价值：用户完成什么任务，减少什么成本，获得什么确定性。
- 成功指标：可量化指标或可观察结果。
- 护栏指标：不能被明显损害的体验、性能、数据质量或运营指标。

## 3. 用户、角色与场景

- 目标用户：主要用户、次要用户、内部角色、外部角色。
- 使用场景：触发时机、入口、设备/平台、频率、上下文。
- 角色权限：查看、创建、编辑、提交、审批、导出、删除、配置、审计。
- 协作关系：谁发起、谁处理、谁确认、谁兜底。

## 4. 范围边界

- 范围内：本次必须解决的问题和能力。
- 非范围：明确不做的事项。
- 延后范围：可以进入后续迭代的事项。
- 模糊边界：需要业务或技术确认的范围。
- 优先级建议：P0/P1/P2，并说明判断理由。

## 5. 功能上下游依赖

检查需求会影响哪些功能链路：

- 上游功能：前置配置、创建入口、审批流程、权限开通、数据录入、外部触发。
- 下游功能：通知、列表、详情、报表、导出、搜索、任务流、运营处理、客服处理。
- 被影响模块：已有页面、已有接口、后台配置、管理端、用户端、消息系统。
- 流程依赖：是否依赖人工审核、定时任务、异步处理、第三方回调。
- 兼容性：老数据、老用户、旧版本客户端、历史流程是否受影响。

## 6. 数据上下游依赖

检查数据从哪里来、到哪里去、如何被使用：

- 数据来源：用户输入、内部系统、第三方系统、埋点、批处理、人工导入。
- 数据写入：写入对象、字段、时机、幂等、失败重试、审计记录。
- 数据消费方：页面展示、搜索筛选、报表、算法、风控、运营、客服、财务。
- 数据口径：字段定义、计算逻辑、时间窗口、去重规则、权限过滤。
- 数据新鲜度：实时、准实时、定时同步、手动刷新。
- 历史数据：迁移、回填、默认值、脏数据、缺失值处理。
- 数据质量：校验、异常监控、对账、一致性要求。

## 7. 状态流转与异常

- 核心对象：需求中最重要的业务对象是什么。
- 状态列表：初始、处理中、成功、失败、取消、过期、归档等。
- 状态变化：谁触发、什么条件触发、是否可逆、是否需要记录。
- 异常情况：空数据、重复提交、权限不足、输入非法、超时、部分成功、第三方失败、数据不一致。
- 兜底机制：重试、撤销、回滚、人工处理、提示文案、日志与告警。

## 8. 业务规则与约束

- 计算规则、排序规则、优先级规则、额度/次数/时间限制。
- 审批规则、风控规则、合规规则、权限规则。
- 跨端、跨地区、跨语言、跨组织的差异。
- 技术、资源、时间、政策或运营约束。

## 9. 埋点、指标与验证

- 成功指标：如何判断需求有效。
- 过程指标：关键漏斗、使用率、完成率、失败率、耗时。
- 事件设计：事件名、触发时机、关键属性。
- 验证方式：数据看板、实验、用户访谈、QA、UAT、运营反馈。

## 10. 发布与运营影响

- 发布方式：灰度、全量、白名单、配置开关。
- 运营准备：SOP、客服话术、后台配置、培训、公告。
- 监控告警：关键指标、异常日志、数据对账。
- 回滚方案：关闭入口、回退配置、数据修复、人工兜底。

## 11. 风险与待确认问题

- 阻塞风险：不解决就不能进入 PRD 或开发。
- 重要风险：可能导致返工、误解、范围扩大或数据错误。
- 假设：当前为了推进而暂时采用的判断。
- 待确认问题：问题、负责人、影响、建议确认时间。
