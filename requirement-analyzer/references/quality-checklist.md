# 需求分析质量检查清单

用于输出前自检，或评审已有需求分析。

## 严重度

- `阻塞 Blocking`：不解决就无法进入 PRD 或开发。
- `重要 Major`：可能造成返工、范围误解、数据错误或原型偏差。
- `次要 Minor`：表达、完整性或协作效率问题。
- `建议 Suggestion`：可选增强。

## 检查项

### 目标与场景

- 是否说清楚为什么要做，而不只是做什么。
- 是否明确目标用户、角色和核心场景。
- 是否区分用户价值和业务目标。
- 是否有成功指标或可观察的成功信号。

### 范围与优先级

- 是否明确范围内、非范围、后续迭代。
- P0/P1/P2 是否有判断依据。
- 是否识别会导致范围扩大的模糊边界。

### 功能依赖

- 是否识别上游入口、配置、权限、审批、触发条件。
- 是否识别下游通知、列表、报表、导出、搜索、运营处理等影响。
- 是否说明对已有模块、老用户、历史流程的兼容影响。

### 数据依赖

- 是否说明数据来源、写入对象、消费方。
- 是否说明字段口径、计算逻辑、同步频率、权限过滤。
- 是否覆盖历史数据、迁移、回填、缺失值、脏数据。
- 是否识别数据质量、对账或监控需求。

### 状态、权限与异常

- 是否定义核心对象和状态流转。
- 是否覆盖不可逆动作、取消、过期、失败、重试、回滚。
- 是否明确不同角色能看什么、做什么。
- 是否覆盖空状态、重复提交、权限不足、第三方失败、数据不一致。

### 交接可用性

- 是否把事实、假设、风险、待确认问题分开。
- 是否给出 PRD 深度建议。
- 是否能被 `prd-generator` 直接用于 PRD 起草。
- 是否指出会影响原型生成的页面、模块、状态或交互。

## 常见修正

- 问题太泛 -> 改成具体场景、用户、任务和触发条件。
- 目标太虚 -> 增加成功指标或可观察结果。
- 依赖遗漏 -> 补上上下游功能、数据源、消费方和影响模块。
- 异常不足 -> 补空状态、错误状态、权限不足、失败重试和人工兜底。
- 待确认问题太多 -> 按阻塞/重要/次要排序，只保留决策相关问题。
