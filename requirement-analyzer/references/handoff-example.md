# 交接示例：requirement-analyzer -> prd-generator

本示例展示需求澄清结果如何交给 `prd-generator`。示例内容为虚构业务场景，只用于说明字段粒度和写法。

# 需求分析：订单异常处理优化

## 1. 需求理解

- **需求来源**：Q3 roadmap 中的核心流程效率主题。
- **核心问题**：订单异常状态分散在支付、库存和履约系统中，用户端说明不清楚，客服后台排查路径长。
- **期望结果**：统一异常状态、补充用户端说明、提供客服后台处理入口和处理记录。
- **当前已知信息**：
  - 支付、库存、履约系统都能返回异常码，但口径尚未统一。
  - 用户端订单详情页和客服后台订单详情页都需要改造。
  - 首版不做异常趋势看板。
- **当前假设**：
  - 上游系统可以在订单详情查询时返回最近一次异常信息。
  - 客服后台具备基础权限体系，可增加高风险动作二次确认。

## 2. 业务目标与成功指标

| 目标 | 成功指标 | 护栏指标 |
| --- | --- | --- |
| 降低用户对异常订单的咨询 | 异常订单相关咨询率下降 | 用户端误导性文案投诉不增加 |
| 降低客服排查成本 | 客服平均处理时长下降 | 客服误操作率不增加 |
| 减少重复处理风险 | 重复退款/重复履约工单下降 | 订单履约成功率不下降 |

## 3. 目标用户与核心场景

| 用户/角色 | 场景 | 任务 | 频率/优先级 |
| --- | --- | --- | --- |
| 普通用户 | 查看异常订单详情 | 理解异常原因和下一步动作 | P0 |
| 客服 | 处理订单异常咨询 | 判断异常来源并执行建议动作 | P0 |
| 运营 | 观察异常类型趋势 | 发现异常峰值和上游问题 | P2 |

## 4. 当前流程与目标流程

- **当前流程**：用户看到订单“处理中/失败” -> 联系客服 -> 客服跨支付、库存、履约系统查询 -> 手动判断处理动作。
- **目标流程**：用户在订单详情页看到明确异常说明和可执行动作；客服在订单详情页看到异常来源、异常码、建议动作和处理记录。
- **主要变化**：异常状态统一映射；用户端和客服端都展示异常上下文；客服操作被记录。

## 5. 需求范围与非范围

| 类型 | 内容 | 说明 |
| --- | --- | --- |
| 范围内 | 异常状态标准化 | 支付、库存、履约异常统一映射到订单状态 |
| 范围内 | 用户端异常说明 | 订单详情页展示原因、建议动作、预计处理时间 |
| 范围内 | 客服后台异常处理区块 | 展示异常来源、建议动作、处理记录 |
| 非范围 | 订单主流程重构 | 不改造下单、支付、履约主链路 |
| 非范围 | 异常趋势看板 | 放入后续迭代 |

## 6. 功能上下游依赖

| 类型 | 依赖/影响对象 | 说明 | 风险 |
| --- | --- | --- | --- |
| 上游 | 支付系统 | 提供扣款失败、支付超时等异常码 | 异常码口径不稳定 |
| 上游 | 库存系统 | 提供锁库存失败、库存不足等异常码 | 可能存在延迟 |
| 上游 | 履约系统 | 提供履约单创建失败等异常码 | 第三方失败信息不完整 |
| 下游 | 用户端订单详情 | 展示异常说明和操作入口 | 文案不清会增加咨询 |
| 下游 | 客服后台订单详情 | 展示处理建议和记录 | 权限不清会导致误操作 |

## 7. 数据上下游依赖

| 类型 | 数据/系统 | 说明 | 待确认 |
| --- | --- | --- | --- |
| 来源 | 支付/库存/履约异常码 | 需要统一异常码、异常来源、发生时间 | 异常码清单 |
| 写入 | 订单异常记录 | 写入异常类型、来源、状态、最近更新时间 | 是否需要历史记录 |
| 消费 | 用户端/客服端/埋点 | 页面展示、客服处理和后续指标分析 | 权限过滤规则 |

## 8. 状态流转与异常情况

- **核心对象**：订单异常记录。
- **状态列表**：支付异常、库存异常、履约异常、未知异常、已处理。
- **状态流转**：上游异常触发 -> 订单异常记录生成/更新 -> 用户端和客服端展示 -> 客服处理 -> 已处理。
- **异常情况**：
  - 上游异常码未知。
  - 同一订单同时出现多个异常。
  - 客服重复提交处理动作。
  - 处理动作失败或超时。
- **兜底机制**：
  - 未知异常展示通用说明和联系客服入口。
  - 高风险动作增加二次确认。
  - 处理失败保留日志并提示人工排查。

## 9. 角色与权限

| 角色 | 可见 | 可操作 | 备注 |
| --- | --- | --- | --- |
| 普通用户 | 自己订单的异常说明 | 重新支付、联系客服 | 不展示技术异常码 |
| 一线客服 | 异常来源、建议动作、处理记录 | 提交低风险处理动作 | 高风险动作需确认 |
| 客服主管 | 全部异常信息和处理记录 | 执行高风险动作 | 需要审计记录 |
| 运营 | 异常类型和处理进度汇总 | 暂无处理动作 | 看板后续迭代 |

## 10. PRD 交接摘要

- **功能摘要**：统一订单异常状态，在用户端展示业务化异常说明，在客服后台提供异常来源、建议动作和处理记录。
- **目标与非目标**：
  - 目标：降低咨询率、缩短客服处理时间、减少重复处理风险。
  - 非目标：不重构订单主流程，不替换上游异常处理逻辑，不做异常趋势看板首版。
- **P0/P1/P2 范围建议**：
  - P0：异常状态标准化、用户端异常说明、客服后台异常处理区块。
  - P1：客服后台异常筛选。
  - P2：异常趋势看板。
- **关键功能依赖**：支付/库存/履约异常码；用户端订单详情；客服后台订单详情；客服权限体系。
- **关键数据依赖**：异常码、异常来源、发生时间、处理动作、处理记录、处理人。
- **关键状态/异常**：支付异常、库存异常、履约异常、未知异常、已处理；多个异常并发；重复提交；处理失败。
- **关键风险**：上游异常码不稳定；用户文案误导；客服误操作。
- **待确认问题**：
  - 上游异常码清单和口径。
  - 预计处理时间是否能给出统一口径。
  - 高风险客服动作有哪些，是否需要主管权限。
- **建议 PRD 深度**：Standard

## 交接给 prd-generator 的提示

生成 PRD 时应重点展开：

- 用户端订单详情和客服后台订单详情两个页面。
- 异常状态映射表。
- 客服处理动作和权限。
- Given/When/Then 验收标准。
- 埋点：异常曝光、用户动作点击、客服处理提交。
