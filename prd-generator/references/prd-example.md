# PRD 示例：订单异常处理优化

这是一个中等复杂度 PRD 示例，用于提供中文 PRD 的写作质感和结构参考。不要机械复制内容；根据用户需求裁剪。

## 1. 需求概述

- **状态**：草稿
- **目标读者**：产品、研发、测试、客服、运营、数据
- **摘要**：当前订单在支付、库存锁定和第三方履约环节出现异常时，用户和客服缺少统一可见的状态说明和处理入口。本需求希望统一订单异常状态、补充用户提示和客服处理动作，降低客服排查成本并提升用户对订单处理进度的确定性。

## 2. 背景与问题

当前订单异常分散在支付系统、库存系统和履约系统中，用户端只看到“处理中”或“失败”，客服后台需要跨多个系统查询。

主要问题：

- 用户不知道订单异常原因和下一步处理方式。
- 客服无法在一个页面查看异常来源、处理状态和建议动作。
- 部分异常缺少明确兜底，导致重复退款、重复创建履约单等风险。

## 3. 目标、指标与非目标

### 目标

| 目标 | 成功信号 |
| --- | --- |
| 提升用户对异常订单状态的理解 | 异常订单相关咨询率下降 |
| 降低客服排查成本 | 客服平均处理时长下降 |
| 减少重复处理风险 | 重复退款/重复履约工单下降 |

### 非目标

- 不重构订单主流程。
- 不替换支付、库存、履约系统的底层异常处理逻辑。
- 不覆盖售后退款之外的复杂争议处理。

### 假设

- 支付、库存、履约系统可以返回标准化异常码。
- 客服后台允许新增异常处理区块。

## 4. 目标用户与核心场景

| 用户/角色 | 场景 | 需求 |
| --- | --- | --- |
| 普通用户 | 查看异常订单详情 | 知道异常原因、预计处理时间和可执行动作 |
| 客服 | 处理用户咨询 | 快速看到异常来源、处理建议和操作入口 |
| 运营 | 监控异常订单 | 查看异常类型分布和处理进度 |

## 5. 范围与优先级

| 功能 | 描述 | 优先级 |
| --- | --- | --- |
| 异常状态标准化 | 将支付、库存、履约异常映射为统一订单状态 | P0 |
| 用户端异常说明 | 在订单详情页展示异常原因和下一步 | P0 |
| 客服后台处理区块 | 展示异常来源、建议动作和处理记录 | P0 |
| 异常订单筛选 | 客服后台支持按异常类型筛选 | P1 |
| 异常趋势看板 | 运营查看异常分布和趋势 | P2 |

## 6. 功能需求

### REQ-001：异常状态标准化

系统必须将支付、库存和履约系统返回的异常码映射为统一订单异常状态。

验收标准：

- Given 支付系统返回扣款失败，当订单详情被查询时，Then 订单状态显示为“支付异常”。
- Given 库存系统返回锁库存失败，当订单详情被查询时，Then 订单状态显示为“库存异常”。
- Given 履约系统返回创建履约单失败，当订单详情被查询时，Then 订单状态显示为“履约异常”。

### REQ-002：用户端异常说明

用户端订单详情页必须展示异常说明、建议动作和预计处理时间。

验收标准：

- Given 订单处于支付异常，当用户进入订单详情页，Then 页面展示失败原因和重新支付入口。
- Given 订单处于库存异常，当用户进入订单详情页，Then 页面展示“库存确认中”和预计处理时间。
- Given 异常状态未知，当用户进入订单详情页，Then 页面展示通用兜底说明和联系客服入口。

### REQ-003：客服后台处理区块

客服后台订单详情页必须展示异常来源、异常码、建议动作和处理记录。

验收标准：

- Given 客服打开异常订单，When 页面加载完成，Then 可见异常来源系统、异常码和最近更新时间。
- Given 异常类型有建议动作，Then 页面展示对应操作入口。
- Given 客服完成处理动作，Then 系统记录处理人、处理时间和动作结果。

## 7. 状态与异常

| 状态 | 来源 | 用户端展示 | 客服端展示 |
| --- | --- | --- | --- |
| 支付异常 | 支付系统 | 支付失败，可重新支付 | 异常码、重试/退款建议 |
| 库存异常 | 库存系统 | 库存确认中 | 锁库存失败原因 |
| 履约异常 | 履约系统 | 商家处理中 | 履约单创建失败原因 |
| 未知异常 | 任一系统 | 处理中，请联系客服 | 原始异常信息和日志链接 |

## 8. 埋点与指标

| 事件/指标 | 触发时机 | 用途 |
| --- | --- | --- |
| order_exception_view | 用户查看异常订单详情 | 衡量异常状态曝光 |
| order_exception_action_click | 用户点击重新支付/联系客服 | 衡量用户自助处理 |
| cs_exception_action_submit | 客服提交处理动作 | 衡量客服处理效率 |
| exception_order_rate | 每日异常订单占比 | 监控异常趋势 |

## 9. 风险与待确认问题

### 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 上游异常码不稳定 | 状态映射错误 | 增加未知异常兜底和异常码监控 |
| 客服误操作 | 重复退款或重复履约 | 高风险动作增加二次确认 |
| 用户提示过于技术化 | 用户仍然无法理解 | 使用业务化文案并让客服参与评审 |

### 待确认问题

| 问题 | 影响 | 建议负责人 |
| --- | --- | --- |
| 三个上游系统能否提供稳定异常码 | 影响状态标准化 | 研发 |
| 预计处理时间是否有统一口径 | 影响用户端展示 | 产品/运营 |
| 哪些客服动作需要二次确认 | 影响后台交互和权限 | 产品/客服负责人 |

2. 与客服确认异常处理动作和权限。
3. 输出订单详情页和客服后台页面原型。
