# PRD 质量检查清单

用于 PRD 评审、优化和交付前的最后检查。

## 评审输出格式

评审时先给问题，按严重程度排序：

- `阻塞 Blocking`：会阻止审批、研发执行或有效验证。
- `重要 Major`：很可能造成返工、歧义、实现偏差或目标遗漏。
- `次要 Minor`：清晰度、一致性、完整性或可维护性问题。
- `建议 Suggestion`：可选优化项。

每条问题包含：

- 影响章节或原文片段
- 问题是什么
- 为什么重要
- 建议怎么改

## 加权评分表

只有当用户要求评分或需要快速质量判断时才打分。默认优先给具体问题。

| 维度 | 权重 | 检查重点 |
| --- | ---: | --- |
| 清晰度 | 20 | 问题、用户、目标、范围、术语是否清楚 |
| 完整性 | 20 | 是否覆盖核心流程、需求、非目标、依赖和风险 |
| 可测试性 | 25 | 验收标准是否可观察、可验证 |
| 优先级 | 15 | P0/P1/P2 或同类优先级是否明确且合理 |
| 执行就绪度 | 15 | 里程碑、负责人、约束、发布和依赖是否足够清楚 |
| 事实可靠性 | 5 | 事实是否有依据，假设是否被标记 |

评分区间：

- `90-100`：基本可进入跨职能确认，只需小修。
- `80-89`：整体可用，但仍有少数关键缺口。
- `70-79`：是有价值的草稿，但执行前需要补充澄清。
- `<70`：暂不适合执行，范围、目标或验收标准需要大幅补强。

## 核心检查项

### 问题与目标

- 问题具体，并且能对应到用户痛点或业务痛点。
- 目标描述的是结果，而不只是“上线某功能”。
- 成功指标可衡量；如果未知，明确标记为待确认。
- 非目标能防止明显的范围蔓延。
- 当可能产生负面影响时，包含护栏指标。

### 用户与范围

- 目标用户和主要场景清楚。
- 需求范围和非范围分开。
- 优先级明确且可信。
- 相关时覆盖角色、权限、平台、地区或语言。
- 依赖和约束可见。

### 需求质量

- 需求描述为可观察行为。
- 重要功能都有验收标准。
- 核心流程覆盖边界情况、空状态、错误处理和恢复路径。
- 相关时覆盖数据状态、生命周期、留存或迁移。
- 非功能需求只写相关内容，不写通用套话。

### 验收与上线

- 验收标准可以通过 QA、UAT、自动化测试、数据看板或运营检查验证。
- 埋点和指标能支持目标判断。
- 当上线有风险时，包含灰度、监控、支持和回滚方案。
- 待确认问题尽量包含影响和下一步负责人。

### 事实与假设

- 市场、竞品、指标、日期、预算、人力、技术判断等内容有来源或被标记为假设。
- 假设没有被包装成事实。
- 会显著影响范围的未知项被放入待确认问题。

## 常见修改方式

- 目标太虚 -> 改成“结果 + 指标 + 时间范围”，或把指标标记为待确认。
- 功能太泛 -> 改成“用户动作 + 系统行为 + 触发条件”。
- 范围太大 -> 拆成 P0 首版范围和后续迭代。
- 缺少验收 -> 为风险最高的流程补 Given/When/Then。
- 依赖隐藏 -> 补充依赖负责人、影响和解决动作。
- 技术细节过多 -> 移到约束或技术方案跟进项。
- 章节是填充内容 -> 删除，或标记“不适用”并说明原因。
