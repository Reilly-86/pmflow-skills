# PRD 结构选择指南

使用本文件选择“刚好够用”的 PRD 结构。不要为了套模板而填充没有决策价值的章节。

## 轻量 PRD

适用场景：小功能、MVP、实验、早期想法、内部工具、需要快速对齐的需求。

推荐结构：

1. 需求概述
2. 问题背景
3. 目标与非目标
4. 目标用户与使用场景
5. 需求范围
6. 功能需求
7. 验收标准
8. 假设与待确认问题
9. 下一步

除非用户明确要求，否则不要写详细技术方案、上线计划或资源计划。

## 标准 PRD

适用场景：常规产品功能、用户可见改动、跨职能协作项目、可进入排期的需求。

推荐结构：

1. 需求概述
2. 背景与问题
3. 目标、成功指标与非目标
4. 目标用户与核心场景
5. 范围与优先级
6. 用户体验与核心流程
7. 功能需求
8. 非功能需求
9. 埋点与数据指标
10. 验收标准
11. 风险、依赖与待确认问题
12. 里程碑与下一步

只有当技术约束会影响产品范围、交付、隐私、安全、性能、集成或数据行为时，才加入技术约束说明。

## 完整 PRD

适用场景：大型项目、多团队协作、强监管场景、重大上线、平台迁移、存在明显交付风险的需求。

推荐结构：

1. 执行摘要
2. 背景、问题与机会
3. 目标、成功指标与护栏指标
4. 用户、角色、场景与任务
5. 范围、非目标、假设与约束
6. 按模块或用户旅程拆分的需求
7. 体验流程、状态、权限与内容规则
8. 数据、埋点、隐私、安全与合规
9. 技术依赖与集成约束
10. 发布、迁移与运营方案
11. 验收策略与测试场景
12. 风险、应对、待确认问题与待决策项
13. 里程碑、负责人和下一步

不要编造负责人、日期、预算或人力配置。缺失时使用“待确认”或放入开放问题。

## 章节选择规则

- 当成功指标、漏斗、实验或用户行为追踪重要时，加入 `埋点与数据指标`。
- 当不同用户有不同权限、管理能力、隐私范围或数据访问范围时，加入 `权限与角色`。
- 当上线、客服支持、监控、迁移、回滚或人工运营重要时，加入 `发布与运营方案`。
- 当 API、数据模型、第三方集成、平台限制或性能约束会影响产品范围时，加入 `技术依赖`。
- 当涉及个人信息、支付、医疗、金融、未成年人、企业数据或监管流程时，加入 `隐私/安全/合规`。
- 当文案、分类、审核、国际化、模板或生成内容影响体验时，加入 `内容规则`。

## 需求写法

优先使用可测试、可观察的需求表达：

```markdown
REQ-001：系统必须允许 [用户] 在 [条件] 下完成 [动作]。
优先级：P0
原因：[为什么重要]
验收标准：
- Given [前置状态]，When [用户/系统行为]，Then [可观察结果]。
- Given [边界情况]，When [用户/系统行为]，Then [安全处理方式]。
```

当用户视角能帮助团队理解价值时，使用用户故事：

```markdown
作为 [用户角色]，
我希望 [能力/行为]，
以便 [获得的结果/价值]。
```
