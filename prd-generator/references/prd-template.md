# PRD 模板

这是 PRD 起草脚手架，不是必须填满的表单。删除没有决策价值的章节；未知事实要标记为“假设”或“待确认问题”。

## 标题

`# PRD：[产品或功能名称]`

## 1. 需求概述

- **状态**：草稿 / 评审中 / 已确认 / 已废弃
- **负责人**：[姓名或角色，未知则写待确认]
- **更新时间**：[日期，必要时填写]
- **目标读者**：产品、设计、研发、测试、数据、运营、客服、管理者
- **摘要**：用 2-4 句话说明要做什么、给谁用、为什么现在要做。

## 2. 背景与问题

说明用户问题、业务背景和当前流程。只有在用户提供或有可靠来源时才写证据，不要编造市场规模、用户数量或竞品结论。

可回答这些问题：

- 用户现在想完成什么任务？
- 当前方案有什么低效、痛苦、风险或缺失？
- 为什么这个问题现在值得解决？
- 如果不做，会产生什么影响？

### 非目标

## 4. 目标用户与核心场景

| 用户类型 | 场景 | 需求 | 频率 / 优先级 |
| --- | --- | --- | --- |
| [用户] | [使用情境] | [用户需要] | [P0/P1/P2 或频率] |

可补充：

- 角色、权限、熟练度、设备/平台、所在地区、限制条件

## 5. 范围与优先级

| 事项 | 描述 | 优先级 | 备注 |
| --- | --- | --- | --- |
| [能力] | [包含什么] | P0/P1/P2 | [约束或依赖] |

优先级建议：

- `P0`：上线或验证价值必须具备
- `P1`：重要，但可以在首版后补充
- `P2`：体验增强或未来迭代

## 6. 用户体验与核心流程

按步骤描述主流程，包括入口、关键决策、状态变化和退出路径。

```markdown
流程：[流程名称]
1. [用户或系统动作]
2. [判断或状态变化]
3. [结果]

边界情况：
- [空状态]
- [错误状态]
- [权限不足]
- [恢复路径]
```

## 7. 功能需求

| ID | 需求 | 优先级 | 原因 |
| --- | --- | --- | --- |
| REQ-001 | 系统必须在 [条件] 下提供 [可观察行为]。 | P0 | [为什么需要] |

复杂需求需要补充验收标准：

```markdown
REQ-001：[需求描述]
验收标准：
- Given [前置条件]，When [操作/事件]，Then [预期结果]。
- Given [边界情况]，When [操作/事件]，Then [处理结果]。
```
